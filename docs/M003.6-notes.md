# M003.6 Notes — Operational Polish & Runtime Corrections

Findings and judgment calls from Mission M003.6 (Phase 2.2,
[roadmap.md](roadmap.md)) — the operational fixes identified during the
M003.5 production validation on RPSG01. Full diagnosis evidence lives in the
operator's investigation log (2026-07-22); this document records what shipped
and what remains external.

## 1. CLI probe discovery under systemd (CLAUDE_BIN / OPENCLAW_BIN)

The agent collector's Claude Code and runtime probes discovered binaries via
`shutil.which()` only. systemd *user* units run with the manager's minimal
`PATH` (`/usr/local/sbin:/usr/local/bin:/usr/sbin:/usr/bin`) — per-user
installs (`~/.local/bin/claude`, `~/.openclaw/tools/node-*/bin/openclaw`)
were invisible, so the monitor showed Claude Code `unavailable` and reported
the **system** `/usr/bin/node` version instead of the Node.js actually
running OpenClaw.

Fix: configurable executable paths (`CLAUDE_BIN`, `OPENCLAW_BIN` in the
agent collector env), with `shutil.which()` PATH discovery preserved as the
fallback when unset. The runtime probe now also reports the `node` binary
**sibling to the configured launcher** (the Node that actually runs
OpenClaw), rendered as `OpenClaw <ver> (<commit>) · node v<ver>`.
A unit drop-in overriding `PATH` was the rejected alternative: env-file
configuration goes through the same reviewed config surface as everything
else and survives unit-file upgrades.

## 2. Persistent journald (host prerequisite; sudo op stays with the operator)

`journalctl --user` returned *"No journal files were opened due to
insufficient permissions"* for the service user — on a `Storage=auto` host
without `/var/log/journal/<machine-id>` the journal is volatile
(`/run/log/journal`, mode `2750`, ACL for group `adm` only) and unreadable
even for the user's own journal; it is also lost on every reboot, which
blocked the M003.5 reboot forensics. The least-privilege fix (persistent
journal directory: `2755` + per-user ACL on `user-<uid>.journal`) is a
**one-time sudo host operation** documented in
[deployment.md](deployment.md) §12 "Persistent journald"; `install.sh` now
warns (without failing) when the journal is unreadable, and the
reboot-recovery checklist gained a journal-readability step. No Observatory
code change beyond docs + the install warning.

**Follow-up finding (operator, 2026-07-22):** when applying the host op on
the fleet's Raspberry Pi it turned out that Raspberry Pi OS **Trixie** ships
a vendor drop-in `/usr/lib/systemd/journald.conf.d/40-rpi-volatile-storage.conf`
with `Storage=volatile`, which overrides `Storage=auto` — so creating
`/var/log/journal` alone is not sufficient on that OS. The fix (applied and
verified live) is an admin drop-in
`/etc/systemd/journald.conf.d/60-observatory-persistent.conf` with
`Storage=persistent` (plus `SystemMaxUse=200M` to cap SD-card usage), which
sorts after and takes precedence over the vendor file. deployment.md §12 and
the `install.sh` warning were updated accordingly.

## 3. Monitor "Last reboot" timezone (DISPLAY_TZ)

`_fmt_last_reboot()` rendered the boot moment — including the
Today/Yesterday day boundary — in UTC, which mislabeled evenings for a
GMT+8 operator (a 20:52 +08 boot showed as "Today 12:52"). New backend
setting `DISPLAY_TZ` (IANA name; default: the host's local timezone from
`/etc/localtime`, honouring `TZ`): wall-clock display values render in that
timezone with an explicit offset suffix, e.g. `Today 20:52 (+08)`. Internal
timestamps, the event stream, and the "generated … UTC" footer stay UTC.
Invalid names log a warning and fall back to the host timezone — a bad
setting must never break the monitor page.

## 4. OpenClaw runtime: dropped post-restart systemEvent (external dependency)

Observed during the M003.5 reboot validation: a cron `systemEvent` wake
scheduled to report after a gateway restart **fired on time but was
degraded** — the OpenClaw runtime created a fresh session whose only user
content was a bare heartbeat poll; the event's reminder text appeared in no
transcript, so the agent answered `HEARTBEAT_OK` and no Telegram report was
sent (the channel itself was healthy: it had come up five minutes earlier
and delivered other traffic later that evening).

This is an **OpenClaw runtime issue, not an Observatory defect — no
Observatory implementation** (mission M003.6 scope ruling). Mitigations on
the operations side:

- post-reboot/post-restart notifications use an **isolated `agentTurn` cron
  job with direct channel delivery**, independent of main-session routing;
- the job follows a **re-arm-until-confirmed** pattern: it reschedules
  itself until the delivery is actually confirmed, so a dropped wake
  degrades to a delayed report instead of silence;
- candidate upstream bug report (dropped systemEvent after gateway
  restart); a possible future Observatory-side complement is an "agent
  silent since boot" alert (Phase 3+ candidate, not scheduled).
