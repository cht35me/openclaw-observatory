# M003.5 Notes — Judgment Calls (PR 2: Host Inventory & Monitor)

Judgment calls made while implementing Mission M003.5 §3–§6 (host inventory,
fleet registry enhancements, monitor improvements), following the
[M003-open-questions.md](M003-open-questions.md) pattern. PR 1 (CI & release
discipline, #5) had no open judgment calls; deployment hardening (§2) and
remaining §7 items follow in PR 3.

## 1. Host Inventory storage: event stream + versioned-row projection

`host_inventory` events flow through the same authenticated pipeline as all
telemetry (durable, auditable record). Additionally, a `host_inventory`
projection table (migration 0004, `ReplacingMergeTree(revision)`, SD-018)
keeps the **latest inventory per host** so reads never scan event history.
The payload is stored as a JSON String, mirroring `events.payload`:
inventory sections are dicts that may gain keys (SMART attributes per
storage device, new hardware families, site metadata) **without schema
changes** — the M003.5 single-node → multi-site scale-out requirement.
Typed columns can be materialized later if query patterns demand them.

## 2. Reduced vs enhanced Fleet & Services view — stored model is central-ready

Per §3e the **local** monitor's Fleet & Services table stays *as is*
(reduced view — unchanged columns). The **enhanced** per-node view is
central-node scope and is deliberately **not** built here. The *stored*
model already carries everything §3e lists for the central renderer:

| §3e field | Where it lives today |
| --- | --- |
| manufacturer, hardware model, revision, installed memory | `host_inventory.payload.hardware` |
| platform | `fleet_registry.platform` |
| operating system, release name, kernel | `host_inventory.payload.os` |
| last heartbeat | derived from the event stream (`FleetAssetView.last_heartbeat`) |
| reboot required, last apt full-upgrade | `host_inventory.payload.maintenance` |
| environment | `fleet_registry.environment` + `DEPLOYMENT_ENVIRONMENT` |

`GET /api/v1/fleet/{id}` + `GET /api/v1/fleet/{id}/inventory` expose the
full model, so the future central node renders the enhanced view without
any schema change.

## 3. Inventory cadence: on start, on durable change, hourly

Host identity is slow-moving; shipping it every 30 s would be noise. The
collector schedules the inventory producer at the fast telemetry interval
but **emits** only when: first run after start, the *identity signature*
changed (signature excludes volatile utilization numbers), or
`INVENTORY_INTERVAL` (default 3600 s) elapsed. The maintenance section
(the only part that shells out) is gathered only when actually emitting.

## 4. Available-updates count: `apt list --upgradable`, no `apt update`

Requirement: best effort **without mutating state or needing root**.
`apt list --upgradable` reads the existing package lists (~1 s on the Pi;
kept fresh by unattended-upgrades) and mutates nothing. The alternative
`apt-get -s upgrade` parse was measured at ~8 s on RPSG01 — too slow for a
collector tick. On failure/timeout the field is honestly `null`. The
freshness of the count is bounded by the last `apt update`, which is itself
reported (`last_apt_update_epoch` from the lists directory mtime).

## 5. apt history: rotation-aware, best-effort latest

`/var/log/apt/history.log` is parsed for the newest `full-upgrade`/
`dist-upgrade` (reported as *last full-upgrade*) and any upgrade activity
including `unattended-upgrade` (reported as *last upgrade*). Rotated
`history.log.N.gz` archives are consulted (newest first) only until both
timestamps are found. A host that never ran a full-upgrade reports `null`.

## 6. Interface relevance filter

`lo`, `veth*`, `br-*`, and `docker*` are excluded: container plumbing says
nothing about *host* connectivity. Physical ports (`eth0`, `wlan0`) and
overlay interfaces (`tailscale0`) are included, with `operstate` rendered
verbatim (tun devices legitimately report `unknown`). IPv4 addresses come
from the stdlib `SIOCGIFADDR` ioctl — no subprocess, no packets.

## 7. Recent Events: heartbeats excluded

§4 lists "heartbeat" among displayable events, but at a 30-second cadence
from three reporters, heartbeats would fill all 20 slots within ~3 minutes
and drown every actually notable event. Judgment: show lifecycle events
(`service_start`, `asset_offline`, `asset_online`, `mission_update`);
heartbeat *absence* is already summarized as offline/online transitions —
stated on the page itself. Collector failures have no discrete event type
(they are heartbeat counters feeding the health score); if a discrete
failure event is introduced later, adding it to `NOTABLE_EVENT_TYPES` is a
one-line change. The backend now records a `service_start` event at each
startup so its own restarts are visible history.

## 8. Build timestamp: committer timestamp of the deployed commit

There is no build pipeline stamping artifacts yet (deployments run from a
git checkout), so §6 "build timestamp" is defined as the **committer
timestamp of the deployed commit**: `BUILD_TIMESTAMP` env override first
(for future packaged builds), else `git log -1 --format=%cI` once at import
— the single deliberate subprocess exception in `app/version.py` (commit
*timestamps* live in packfiles, unlike the commit *SHA* which is still read
stdlib-only) — else an honest `unknown`.

## 9. Environment classification: Development unless stated

`DEPLOYMENT_ENVIRONMENT` (backend) and `FleetAsset.environment` (per asset)
default to **Development**: production status is an explicit operator/seed
statement, never an accident. Seed entries for RPSG01/A001/OBLN01 declare
`Production` (they are tagged `production` in FLEET.md). Note: seeding is
create-only, so registries seeded before M003.5 keep `Development` until
administered — honest, and PR 3's upgrade procedure is the natural place to
set the live values (`DEPLOYMENT_ENVIRONMENT=Production` in the backend env
file).

## 10. Last reboot rendering

Derived from `system_metrics.uptime_seconds` (no new probe):
`Today HH:MM` / `Yesterday HH:MM` / `N days ago`, in UTC like the page's
`generated` stamp. The spec's "Never" case cannot occur for a booted Linux
host reporting uptime; an absent uptime renders `—`.
